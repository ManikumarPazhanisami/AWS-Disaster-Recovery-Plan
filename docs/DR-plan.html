Failover Execution Timeline

⚠️
Execute this plan only when a disaster has been formally declared by the Incident Commander. Coordinate all steps with the DR team before proceeding.
T + 0 min
Incident Declared
On-call engineer detects outage. Incident Commander activated. DR team assembled.
T + 5 min
Aurora Replication Verified
Check replication lag in CloudWatch. Verify DR cluster status is "available".
T + 7 min
Aurora Promoted to Primary
Promote ias-prod-cluster-dr in us-west-1. Cluster becomes writable. ~2 min promotion time.
T + 10 min
EC2 Instances Launched
Launch app servers and Nginx proxy from pre-built Launch Templates. Wait for healthy status.
T + 15 min
DNS Updated
Update Route53 A record and DB CNAME to point to us-west-1 endpoints. TTL: 60s.
T + 20 min
DNS Propagated
Wait for TTL expiry + propagation (~3 min). Verify with dig api.yourdomain.com.
T + 30 min
✅ Application Live in DR
End-to-end health check passes. Services restored. Notify customers. Begin monitoring.

Appendix D: Troubleshooting Guide

High Aurora Replication Lag
AuroraGlobalDBReplicationLag > 5 seconds
Check network connectivity between regions. Increase source DB capacity. Review cross-region bandwidth limits.
S3 Replication Slow
Objects not in DR bucket within 15 min
Verify CRR rules enabled. Check IAM role permissions for the replication role. Review S3 CRR metrics in console.
EC2 AMI Copy Failed
AMI copy status shows "failed"
Re-run copy command. Check that KMS key is accessible cross-region. Verify destination region permissions.
Route53 Health Check Failing
Health check fails but primary is healthy
Ensure security groups allow traffic from Route53 health checker IPs. Verify /health endpoint returns HTTP 200.
Lambda VPC Connection Error
Lambda cannot connect to RDS
Verify Lambda ENI is in private subnet. Check security group rules allow egress on 3306. Confirm RDS SG allows ingress from sg-lambda.
SES Email Bouncing
Emails bounce in DR region
Verify domain is verified in us-west-1 SES. Check DKIM/SPF/DMARC DNS records. Request production access if still in sandbox mode.
DNS Not Updating
dig shows old IP after failover
Wait for TTL (60s) + propagation (2–5 min). Flush DNS cache: sudo systemctl restart systemd-resolved
DB Connection Pool Exhausted
Connection errors in application logs
Scale up Aurora ACUs (Serverless v2 will auto-scale to 8 ACU max). Tune connection pooling. Increase max_connections.
